# 框架运维排障手册（简版）

本文档面向 `zMmoServer` 当前框架实现，重点说明跨服通信、去重和一致性相关的排障入口。

## 一、先看哪些指标

GameServer 指标（`/metrics`）优先关注：

- `game_outbox_pending`
  - 含义：待重试的跨服消息数量
  - 异常信号：持续上升且不回落
- `game_outbox_dead`
  - 含义：死信消息数量（超过重试上限或元数据异常）
  - 异常信号：大于 0 且持续增长
- `game_gateway_dedupe_hits_total`
  - 含义：Gateway 请求去重命中累计次数
  - 异常信号：短时间突增，可能存在重复投递或重放

辅助指标：

- `game_connections_current`
- `game_sessions_current`
- `game_players_online`

## 二、关键日志关键词

### 1) 去重相关

- `Duplicate gateway request ignored`
  - 查看字段：`request_id`、`proto_id`
  - 说明：请求被幂等去重拦截，业务未重复执行

### 2) Outbox 相关

- `Cross-server send failed`
  - 查看字段：`trace_id`、`request_id`、`proto_id`、`map_id`
- `Outbox message moved to dead-letter`
  - 查看字段：`request_id`、`attempts`、`topic`
- `Outbox dead-letter snapshot`
  - 查看字段：`request_id`、`topic`、`attempts`、`last_error`

### 3) Envelope 相关

- `Received cross-server envelope from Gateway`
- `Received cross-server envelope from MapServer`
- `Received cross-server envelope from GameServer`
- `Sending cross-server envelope to GameServer`

字段统一关注：

- `trace_id`
- `request_id`
- `proto_id`

## 三、排障流程建议

1. 先看 `game_outbox_dead` 是否 > 0  
2. 若 > 0，定位 `Outbox dead-letter snapshot` 对应 `request_id`  
3. 用同一 `request_id` 在 Gateway/Game/Map 日志中串联查询  
4. 同时观察 `game_outbox_pending` 是否长期堆积  
5. 若 `game_gateway_dedupe_hits_total` 快速增长，排查 Gateway 到 Game 的重发策略或网络抖动

## 四、常见问题与建议

- 死信持续增长：
  - 优先检查 MapServer 可达性和 `map_ids` 映射
  - 检查 `proto_id` 与路由处理器是否匹配
- 去重命中异常高：
  - 检查 Gateway 是否重复发送同一业务请求
  - 检查连接断线重连后的重放策略
- Pending 长期不降：
  - 可能 MapServer 响应链路存在堵塞
  - 建议联查 `Received map attack response` 日志频率

## 五、当前框架能力边界

目前实现为“框架级最小闭环”：

- 已支持：request_id 去重、Outbox 重试、DLQ、跨服追踪日志
- 未实现：持久化 Outbox/Inbox（当前为内存实现）、完整管理 API

后续如需线上强保障，建议将 Outbox/Inbox 落到持久化存储。

## 2. 目录功能说明

### 2.1 核心服务器目录 (`*/global/`, `*/gateway/`, `*/game/`, `*/maps/`, `*/admin/`)
- **功能**：包含服务器的核心实现，继承自 `zMmoShared/server` 中的基础服务器
- **文件结构**：
  - `base_server.go`：服务器的核心实现，包含生命周期管理和组件注册
  - 其他相关文件：根据服务器类型的特定需求添加

### 2.2 配置管理目录 (`*/config/`)
- **功能**：负责加载和管理服务器配置
- **文件结构**：
  - `config.go`：配置结构体定义和加载逻辑
  - 其他配置相关文件

### 2.3 连接管理目录 (`*/connection/`)
- **功能**：管理客户端连接和会话
- **文件结构**：
  - `connection_manager.go`：连接管理器实现
  - 其他连接相关文件

### 2.4 服务实现目录 (`*/service/`)
- **功能**：实现具体的业务服务
- **文件结构**：
  - `tcp_service.go`：TCP服务实现
  - 其他服务相关文件

### 2.5 共享包目录 (`zMmoShared/`)
- **功能**：提供各服务器共享的功能和组件
- **文件结构**：
  - `server/`：服务器框架
  - `config/`：配置管理
  - `protocol/`：协议定义
  - 其他共享功能

### 2.6 资源文件目录 (`resources/`)
- **功能**：存放配置表、地图文件等资源
- **文件结构**：
  - `excel_tables/`：Excel格式的配置表
  - `maps/`：地图文件
  - `protocol/`：协议文件

### 2.7 文档目录 (`docs/`)
- **功能**：存放项目文档
- **文件结构**：
  - `架构设计方案.md`：系统架构设计
  - `架构分析与改进建议.md`：架构分析和改进建议
  - `统一目录结构规范.md`：目录结构规范
  - 其他文档

## 3. 代码组织规范

### 3.1 服务器实现规范
1. **继承基础服务器**：所有服务器必须继承自 `zMmoShared/server.BaseServer`
2. **实现生命周期方法**：重写 `onBeforeStart`, `onAfterStart`, `onBeforeStop` 等生命周期方法
3. **组件注册**：使用 `RegisterComponent` 方法注册组件
4. **统一启动流程**：使用 `server.ServerTemplate` 启动服务器

### 3.2 目录命名规范
1. **目录名**：使用小写单词，不使用下划线或驼峰
2. **文件名**：使用小写蛇形命名（如 `base_server.go`）
3. **包名**：使用小写单词，与目录名保持一致

### 3.3 代码风格规范
1. **命名规范**：
   - 导出函数/类型：大写驼峰
   - 私有函数/类型：小写驼峰
   - 接口：以 `er` 结尾或 `I` 前缀
   - 常量：大写驼峰或全大写下划线分隔

2. **格式规范**：
   - 使用 `gofmt` 或 `goimports` 格式化代码
   - 缩进使用 Tab
   - 行长度不超过 120 字符
   - 大括号不换行

3. **注释规范**：
   - 包注释：写在 `package` 语句上方
   - 导出函数/类型：必须添加注释
   - 私有函数/类型：复杂逻辑添加注释
   - 行内注释：解释复杂或非直观的代码

## 4. 服务器启动流程

### 4.1 启动流程
1. **创建基础服务器**：调用 `NewBaseServer()` 创建服务器实例
2. **创建启动模板**：调用 `server.NewServerTemplate()` 创建启动模板
3. **设置回调函数**：根据需要设置 `SetupFunc`, `StartFunc`, `ShutdownFunc`
4. **运行服务器**：调用 `template.Run()` 启动服务器

### 4.2 生命周期
1. **初始化**：加载配置，初始化组件
2. **启动**：启动各组件服务
3. **运行**：处理业务逻辑
4. **停止**：优雅关闭各组件

## 5. 组件管理

### 5.1 组件注册
使用 `RegisterComponent(name string, component interface{})` 方法注册组件，便于后续通过 `GetComponent(name string)` 获取组件实例。

### 5.2 组件命名规范
- 使用清晰、描述性的名称
- 避免使用缩写和歧义名称
- 保持组件名称的一致性

## 6. 配置管理

### 6.1 配置文件位置
- 配置文件应放置在服务器根目录下，命名为 `config.ini`
- 配置表文件应放置在 `resources/excel_tables/` 目录

### 6.2 配置加载
- 使用统一的配置加载函数 `config.LoadConfig()`
- 配置加载应在服务器启动前完成
- 配置验证应在加载后进行

## 7. 日志管理

### 7.1 日志初始化
- 使用 `zLog.InitLogger()` 初始化日志
- 日志文件应放置在 `./logs/` 目录
- 日志级别应根据环境设置

### 7.2 日志使用
- 使用 `zLog.Info()`, `zLog.Error()`, `zLog.Fatal()` 等方法记录日志
- 日志应包含足够的上下文信息
- 避免在日志中记录敏感信息

## 8. 错误处理

### 8.1 错误传递
- 错误应向上传递，不被忽略
- 错误传递时应添加上下文信息
- 使用 `fmt.Errorf("...: %w", err)` 包装错误

### 8.2 错误处理
- 服务器启动时的错误应导致启动失败
- 运行时错误应被捕获并记录，不影响服务器运行
- 关键错误应触发告警

## 9. 最佳实践

### 9.1 代码组织
- 每个文件应保持单一职责
- 函数应保持简短，不超过 50 行
- 避免重复代码，提取公共功能到共享包

### 9.2 性能优化
- 避免频繁的内存分配
- 使用对象池复用对象
- 合理使用并发

### 9.3 安全性
- 验证所有用户输入
- 防止 SQL 注入和 XSS 攻击
- 敏感数据应加密存储

### 9.4 可维护性
- 编写清晰的注释和文档
- 使用一致的命名和代码风格
- 编写单元测试和集成测试

## 10. 总结

统一的目录结构和代码组织规范有助于提高代码的可读性、可维护性和可扩展性。所有开发人员应严格遵循本规范，确保项目代码的一致性和质量。

随着项目的发展，本规范可能会根据实际需求进行调整和完善。

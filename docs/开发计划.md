根据项目需求，本次开发遵循以下约束：

| 编号 | 约束 | 影响 |
|------|------|------|

| C2 | 优化 zUtil 不足之处 | 需修改 zUtil 源码 |

## 二、版本冲突分析与统一方案

### 2.1 当前版本冲突清单

| 依赖库 | 冲突描述 | 涉及模块 |
|--------|----------|----------|
| **protobuf** | zEngine/GlobalServer 用 `replace => v1.36.5`，其余用 `v1.36.10` | zEngine, GlobalServer |
| **JWT** | GlobalServer 用 `jwt v3.2.2+incompatible`，GatewayServer 用 `jwt/v5 v5.2.0` | GlobalServer, GatewayServer |
| **Redis** | zCommon 用 `go-redis/redis/v8`，应升级到 `redis/go-redis/v9` | zCommon |
| **gogo/protobuf** | GatewayServer/GameServer 直接依赖（已归档库） | GatewayServer, GameServer |
| **golang/protobuf** | MapServer 直接依赖旧版 `golang/protobuf v1.5.4` | MapServer |
| **golang/snappy** | zEngine 用 `v1.0.0`，zCommon 用 `v0.0.4` | zEngine, zCommon |

### 2.2 版本统一方案

| 依赖库 | 目标版本 | 操作 |
|--------|----------|------|
| `google.golang.org/protobuf` | **v1.36.10** | 移除 zEngine/GlobalServer 的 replace 指令，统一使用 v1.36.10 |
| `golang-jwt/jwt` | **v5.2.0** | GlobalServer 从 v3 迁移到 v5，API 有变化需适配 |
| `redis/go-redis/v9` | **v9.x** | zCommon 从 v8 升级到 v9，API 变化需适配（NewClient 参数等） |

| `golang/protobuf` | **移除直接依赖** | MapServer 改用 `google.golang.org/protobuf` |
| `golang/snappy` | **v1.0.0** | zCommon 从 v0.0.4 升级到 v1.0.0 |

### 阶段二：zEngine 优化

> 修复评估报告中识别的 zEngine 不足之处

#### 任务 2.1：修复 DDoS 防护

- **问题**：`zNet/ddosProtection.go` 的 `AllowConnection()` 无条件返回 true
- **方案**：实现基于令牌桶的连接频率限制
  - 可配置的连接速率阈值（每秒最大新连接数）
  - 可配置的 IP 级连接数上限
  - 白名单机制（内网 IP 默认放行）
  - 统计信息暴露给 Prometheus

#### 任务 2.2：Actor 监督机制

- **问题**：`zActor` 缺少 Actor 重启/监督机制，Actor panic 后无法恢复
- **方案**：
  - 添加 `SupervisorStrategy` 枚举（Restart/Stop/Escalate）
  - `BaseActor` 添加 panic 恢复 + 自动重启逻辑
  - 添加重启次数限制和退避策略
  - 添加 Actor 生命周期回调（OnRestart/OnStop）
  - 添加 `Supervisor` 接口管理子 Actor

#### 任务 2.3：分布式锁续期机制

- **问题**：`zDistributed/etcd_lock.go` 缺少锁续期（Watchdog）机制，长时间操作可能丢锁
- **方案**：
  - 添加自动续期 goroutine（Watchdog）
  - 基于 etcd Lease 的 KeepAlive 实现续期
  - 可配置续期间隔（默认 TTL 的 1/3）
  - 支持 context 取消时自动停止续期并释放锁
  - 添加 `LockWithAutoRenew` 方法

#### 任务 2.4：ObjectManager 泛型化

- **问题**：`zObject/ObjectManager` 使用 `interface{}` 作为 ID 类型，类型安全性不足
- **方案**：
  - 使用 Go 泛型重构 `ObjectManager[ID, Object]`
  - ID 类型约束为 `comparable`
  - 提供类型安全的 `Get(id ID) (Object, bool)` 等方法
  - 保留旧的 `ObjectManager` 作为 `ObjectManagerAny` 兼容

#### 任务 2.5：zNet 加密开关

- **问题**：服务器间通信不需要加密，当前 zNet 加密不可配置
- **方案**：
  - `TcpServer` 添加 `EnableEncryption bool` 配置项
  - 服务器间连接创建时设置 `EnableEncryption = false`
  - 客户端连接保持 `EnableEncryption = true`
  - 跳过加密时同时跳过 ECDH 密钥交换和密钥轮换
  - 跳过压缩选项也独立可配置（小消息不压缩更高效）

#### 任务 2.6：zNet 消息优先级支持

- **问题**：即时游戏需要消息优先级，当前所有消息平等处理
- **方案**：
  - 定义消息优先级枚举（High/Normal/Low）
  - 高优先级：移动、战斗、技能释放
  - 普通优先级：聊天、交易、UI 操作
  - 低优先级：日志、统计、心跳
  - `TcpServer` 发送队列改为优先级队列
  - 高优先级消息立即刷新缓冲区

#### 任务 2.7：zActor 消息优先级

- **问题**：Actor 的 channel 消息无优先级，紧急消息可能被排队延迟
- **方案**：
  - `BaseActor` 添加优先级消息通道
  - 处理循环中优先检查高优先级通道
  - 添加 `SendPriority(msg, priority)` 方法
  - 保持与现有 `Send()` 方法的兼容性

### 阶段三：zUtil 优化

> 修复评估报告中识别的 zUtil 不足之处

#### 任务 3.1：zRand 安全随机

- **问题**：`zRand/rand.go` 使用 `math/rand` + `time.Now().UnixNano()` 种子，不安全且不必要
- **方案**：
  - Go 1.20+ 已自动种子，移除手动种子代码
  - 游戏逻辑随机使用 `math/rand/v2`（Go 1.22+）
  - 安全随机（Token/密钥生成）使用 `crypto/rand`
  - 添加 `SecureInt`/`SecureString` 等安全随机方法

#### 任务 3.2：zCache LRU 退出机制

- **问题**：`zCache/zCache.go` LRU 的 `cleanup()` goroutine 没有退出机制，会导致 goroutine 泄漏
- **方案**：
  - 添加 `context.Context` 参数到 `NewLRUCache`
  - `cleanup()` goroutine 监听 context 取消信号
  - 添加 `Close()` 方法取消 context 并等待 goroutine 退出
  - 添加 `runtime.SetFinalizer` 兜底保护

#### 任务 3.3：zHashtable 并发安全

- **问题**：`zHashtable/zHashtable.go` 自动扩容在并发环境下可能不安全
- **方案**：
  - 扩容操作加读写锁保护
  - 扩容期间使用双缓冲策略（新旧表并存，读旧写新）
  - 扩容完成后原子切换表指针
  - 添加并发测试验证

#### 任务 3.4：zQueue 环形缓冲区

- **问题**：`zQueue/queue.go` 基于链表实现，性能不如环形缓冲区
- **方案**：
  - 新增 `RingQueue` 基于环形缓冲区实现
  - 预分配固定容量，避免频繁内存分配
  - 保留 `LinkedQueue` 用于容量不确定的场景
  - 添加基准测试对比性能

### 阶段四：zCommon 代码去重与重构

> 消除各服务间的重复代码，提升可维护性

- **问题**：`getConfigString/Int/Bool` 在 Global/Gateway/Map 三个服务中重复定义

  - 提供通用的 `GetConfigString/Int/Bool/Float` 函数
  - 各服务删除本地重复定义，改为导入 zCommon

#### 任务 4.2：提取版本信息

- **问题**：`version.go` 在所有服务中完全相同
- **方案**：
  - 在 `zCommon/common/` 下新增 `version.go`
  - 各服务删除本地 `version.go`，改为导入 zCommon

#### 任务 4.3：提取健康检查

- **问题**：`health/checker.go` 在 Game/Map 中几乎相同
- **方案**：
  - 在 `zCommon/health/` 下统一健康检查实现
  - 支持自定义检查项注册
  - 各服务删除本地重复代码

#### 任务 4.4：提取请求去重与超时

- **问题**：`request/dedupe.go` 和 `request/timeout.go` 在 Game/Map 中完全相同
- **方案**：
  - 在 `zCommon/request/` 下统一实现
  - 各服务删除本地重复代码

#### 任务 4.5：提取通用工具函数

- **问题**：`utils/error.go` 和 `utils/retry.go` 在 Game/Map 中完全相同
- **方案**：
  - 在 `zCommon/utils/` 下统一实现
  - 各服务删除本地重复代码

#### 任务 4.6：统一服务发现实现

- **问题**：`discovery/service_discovery.go` 在 Gateway/Game/Map 中各自实现，逻辑高度相似
- **方案**：
  - 在 `zCommon/discovery/` 下统一服务发现实现
  - 提供通用的注册/发现/心跳/Watch 逻辑
  - 各服务只需提供自身配置即可使用

#### 任务 4.7：简化 DAO/Repository 模式

- **问题**：DAO 层回调模式过于复杂，同步调用需要 channel 等待；Async/Sync 双套方法导致代码量翻倍
- **方案**：
  - 移除回调模式，改为直接数据库操作（连接池已保证并发安全）
  - 统一使用异步方法，需要同步时由调用方自行等待
  - 减少代码量约 40%

### 阶段五：架构基础设施完善

> 聚焦通信、服务发现、数据一致性、内存、线程模型、数据存储、资源管理

#### 任务 5.1：服务器间通信优化

- **目标**：最大化服务器间消息传输速度
- **方案**：
  - 利用任务 2.5 的加密开关，服务器间通信禁用加密
  - 小消息（< 256 字节）禁用压缩，减少 CPU 开销
  - 服务器间连接使用更大的 TCP 缓冲区（64KB）
  - 启用 TCP_NODELAY，禁用 Nagle 算法
  - 消息批量发送：同一帧内的多个消息合并发送，减少系统调用
  - 添加消息发送统计（延迟 P50/P95/P99）

#### 任务 5.2：服务发现优化

- **目标**：服务发现快速响应，减少服务上线/下线的感知延迟
- **方案**：
  - 利用任务 4.6 的统一服务发现实现
  - 优化 etcd Watch 的最小修订版追踪，避免历史事件重放
  - 添加本地缓存 + Watch 增量更新，减少 etcd 查询
  - 服务下线时主动注销（优雅关闭），而非仅依赖 TTL 过期
  - 添加服务健康状态标记（Starting/Ready/Draining）

#### 任务 5.3：数据一致性框架

- **目标**：跨服操作的数据一致性保障
- **方案**：
  - 完善 `zCommon/consistency/` 的 Inbox/Outbox 机制
  - 添加消息持久化选项（可选 Redis 或文件持久化）
  - 实现至少一次投递保证（At-Least-Once Delivery）
  - 添加消息去重（基于 Envelope 的 MessageID）
  - 实现跨服操作的两阶段提交框架（用于跨服地图进入/退出）
  - 添加操作日志（Operation Log）用于故障恢复

#### 任务 5.4：内存管理

- **目标**：减少 GC 压力，避免频繁内存分配
- **方案**：
  - 在 `zCommon` 下新增 `pool/` 包，提供通用对象池
  - NetPacket 对象池化（`sync.Pool`）
  - Protobuf 消息对象池化
  - 跨服信封对象池化
  - 地图对象（玩家/怪物/NPC）预分配
  - 添加内存使用监控指标

#### 任务 5.5：线程模型优化

- **目标**：明确各服务的 goroutine 模型，减少锁竞争
- **方案**：
  - 明确 Actor 模型的使用边界：
    - Player 对象 → Actor 模型（串行化玩家操作）
    - Map 对象 → Actor 模型（串行化地图内操作）
    - 全局服务 → 传统并发（锁/通道）
  - 利用任务 2.7 的 Actor 消息优先级，确保移动/战斗消息优先处理
  - Map 内部分区域并行更新（将大地图分为多个区域，每个区域一个 goroutine）
  - 添加 goroutine 数量监控和告警

#### 任务 5.6：数据存储层

- **目标**：完善数据存取架构（不做数据库本身优化）
- **方案**：
  - 简化 DAO/Repository 层（利用任务 4.7）
  - 实现玩家数据的完整 CRUD
  - 实现玩家上线加载/下线保存流程
  - 添加数据版本号，支持乐观锁
  - 实现定时自动存档机制
  - 添加存档失败重试

#### 任务 5.7：资源管理

- **目标**：配置表和地图资源的加载与管理
- **方案**：
  - 完善配置表加载验证（启动时检查必需配置表）
  - 添加配置表版本号管理
  - 地图资源加载优化（按需加载，而非全量加载）
  - 资源引用计数，支持卸载不用的地图资源
  - 添加资源加载状态监控

每个阶段完成后需验证：

- [ ] 所有模块编译通过（`go build ./...`）
- [ ] 无静态检查错误（`golangci-lint run`）
- [ ] 现有功能不受影响（手动测试客户端连接）
- [ ] go.mod/go.sum 正确（`go mod tidy`）

### 7.3 风险提示

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| JWT v3→v5 迁移 API 不兼容 | GlobalServer 认证功能可能中断 | 先在测试环境验证，保留 v3 回退方案 |
| Redis v8→v9 API 变化 | zCommon 数据库操作可能失败 | 逐模块迁移，每步验证 |
| Protobuf 版本统一可能暴露隐藏问题 | 序列化/反序列化行为变化 | 全量回归测试 |
| zEngine 修改影响所有服务 | 修改不当可能导致全项目编译失败 | 每次修改后全项目编译验证 |
| Actor 监督机制引入复杂度 | 可能引入新的并发 bug | 充分测试，渐进式启用 |

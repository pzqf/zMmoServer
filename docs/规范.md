### 1.2 目录功能说明

#### 1.2.1 核心服务器目录 (`*/global/`, `*/gateway/`, `*/game/`, `*/maps/`, `*/admin/`)
- **功能**：包含服务器的核心实现，继承自 `zCommon/server` 中的基础服务器
- **文件结构**：
  - `base_server.go`：服务器的核心实现，包含生命周期管理和组件注册
  - 其他相关文件：根据服务器类型的特定需求添加

#### 1.2.2 配置管理目录 (`*/config/`)
- **功能**：负责加载和管理服务器配置
- **文件结构**：
  - `config.go`：配置结构体定义和加载逻辑
  - 其他配置相关文件

#### 1.2.3 连接管理目录 (`*/connection/`)
- **功能**：管理客户端连接和会话
- **文件结构**：
  - `connection_manager.go`：连接管理器实现
  - 其他连接相关文件

#### 1.2.4 服务实现目录 (`*/service/`)
- **功能**：实现具体的业务服务
- **文件结构**：
  - `tcp_service.go`：TCP服务实现
  - 其他服务相关文件

#### 1.2.5 共享包目录 (`zCommon/`)
- **功能**：提供各服务器共享的功能和组件
- **文件结构**：
  - `server/`：服务器框架
  - `config/`：配置管理
  - `protocol/`：协议定义
  - 其他共享功能

#### 1.2.6 资源文件目录 (`resources/`)
- **功能**：存放配置表、地图文件等资源
- **文件结构**：
  - `excel_tables/`：Excel格式的配置表
  - `maps/`：地图文件
  - `protocol/`：协议文件

#### 1.2.7 文档目录 (`docs/`)
- **功能**：存放项目文档
- **文件结构**：
  - `架构设计方案.md`：系统架构设计
  - `架构分析与改进建议.md`：架构分析和改进建议
  - `统一目录结构规范.md`：目录结构规范
  - 其他文档

### 1.3 代码组织规范

#### 1.3.1 服务器实现规范
1. **继承基础服务器**：所有服务器必须继承自 `zCommon/server.BaseServer`
2. **实现生命周期方法**：重写 `onBeforeStart`, `onAfterStart`, `onBeforeStop` 等生命周期方法
3. **组件注册**：使用 `RegisterComponent` 方法注册组件
4. **统一启动流程**：使用 `server.ServerTemplate` 启动服务器

#### 1.3.2 目录命名规范
1. **目录名**：使用小写单词，不使用下划线或驼峰
2. **文件名**：使用小写蛇形命名（如 `base_server.go`）
3. **包名**：使用小写单词，与目录名保持一致

#### 1.3.3 代码风格规范
1. **命名规范**：
   - 导出函数/类型：大写驼峰
   - 私有函数/类型：小写驼峰
   - 接口：以 `er` 结尾或 `I` 前缀
   - 常量：大写驼峰或全大写下划线分隔

2. **格式规范**：
   - 使用 `gofmt` 或 `goimports` 格式化代码
   - 缩进使用 Tab
   - 行长度不超过 120 字符
   - 大括号不换行

3. **注释规范**：
   - 包注释：写在 `package` 语句上方
   - 导出函数/类型：必须添加注释
   - 私有函数/类型：复杂逻辑添加注释
   - 行内注释：解释复杂或非直观的代码

### 1.4 服务器启动流程

#### 1.4.1 启动流程
1. **创建基础服务器**：调用 `NewBaseServer()` 创建服务器实例
2. **创建启动模板**：调用 `server.NewServerTemplate()` 创建启动模板
3. **设置回调函数**：根据需要设置 `SetupFunc`, `StartFunc`, `ShutdownFunc`
4. **运行服务器**：调用 `template.Run()` 启动服务器

#### 1.4.2 生命周期
1. **初始化**：加载配置，初始化组件
2. **启动**：启动各组件服务
3. **运行**：处理业务逻辑
4. **停止**：优雅关闭各组件

### 1.5 组件管理

#### 1.5.1 组件注册
使用 `RegisterComponent(name string, component interface{})` 方法注册组件，便于后续通过 `GetComponent(name string)` 获取组件实例。

#### 1.5.2 组件命名规范
- 使用清晰、描述性的名称
- 避免使用缩写和歧义名称
- 保持组件名称的一致性

## 二、配置文件管理规范

### 2.1 配置文件格式与位置

#### 2.1.1 配置表文件

**位置**: `resources/excel_tables/` 目录
**格式**: Excel (.xlsx)
**文件命名**: 小写蛇形命名

| 配置项 | 文件名 | 内容 |
|--------|--------|------|
| 技能配置 | skill.xlsx | 技能名称、伤害值、冷却时间等 |
| 物品配置 | item.xlsx | 物品名称、类型、价格等 |
| 宠物配置 | pet.xlsx | 宠物名称、属性、技能等 |
| 坐骑配置 | mount.xlsx | 坐骑名称、属性、速度等 |
| 任务配置 | quest.xlsx | 任务条件、奖励、流程等 |
| 商店配置 | shop.xlsx | 商店名称、物品、价格等 |
| 成就配置 | - | 成就条件、奖励、难度等 |
| 地图配置 | map.xlsx | 地图名称、类型、场景文件等 |
| 怪物配置 | monster.xlsx | 怪物名称、属性、AI、掉落等 |
| 刷新点配置 | spawn_point.xlsx | 怪物刷新点位置、类型、数量等 |
| 玩家等级 | player_level.xlsx | 等级经验值、属性成长等 |
| BUFF配置 | buff.xlsx | BUFF效果、持续时间等 |
| 公会配置 | guild.xlsx | 公会等级、技能、建筑等 |
| AI配置 | ai.xlsx | AI行为模式、参数等 |

#### 2.1.2 地图文件

**位置**: `resources/maps/` 目录
**格式**: JSON
**文件命名**: 地图名称.json

#### 2.1.3 协议文件

**位置**: `resources/protocol/` 目录
**格式**: Protocol Buffers (.proto)

### 2.2 代码加载规范

#### 2.2.1 配置表加载

- 使用 `zCommon/config/tables/` 中的表格管理类加载配置
- 加载顺序: 启动时一次性加载所有配置表
- 缓存策略: 配置加载后缓存到内存，避免重复读取

#### 2.2.2 地图文件加载

- 使用 `zCommon/config/tables/map_table.go` 加载地图配置
- 地图数据根据地图 ID 索引，便于快速查找

#### 2.2.3 配置更新

- 配置表更新后需要重启服务器才能生效
- 地图文件更新后需要重新加载地图数据

### 2.3 注意事项

1. **禁止在其他目录创建配置文件**
   - 所有配置文件必须放在指定目录
   - 禁止在 `MapServer/config/` 等目录创建JSON配置文件

2. **配置格式要求**
   - Excel表格必须有正确的表头
   - JSON文件必须符合语法规范
   - 所有配置项必须有默认值

3. **配置版本管理**
   - 配置文件更新需要记录版本号
   - 重大变更需要在更新日志中说明

4. **错误处理**
   - 配置加载失败时必须记录详细错误
   - 关键配置缺失时服务器应拒绝启动

5. **性能优化**
   - 配置加载应在启动时完成，避免运行时IO操作
   - 频繁访问的配置应使用缓存

### 2.4 迁移步骤

#### 2.4.1 从旧配置迁移到新配置

1. **停止服务器**
2. **备份旧配置**
3. **将配置转换为Excel格式**
4. **放入 `resources/excel_tables/` 目录**
5. **修改代码加载逻辑**
6. **启动服务器并验证**

#### 2.4.2 配置验证

启动时验证以下内容:
- 所有必要的配置表是否存在
- 配置格式是否正确
- 配置项是否完整
- 配置间关系是否正确

### 2.5 ServerID 配置规范

#### 2.5.1 ServerID 格式（Gateway/Game/Map）

- 使用6位数字 ServerID，格式为 GroupID(4) + ServerIndex(2)
- 例如：组1的第1个服务器，ServerID 为 `000101`
- 在 etcd 中的 `metadata.server_id` 必须存储为 `000101` 这样的6位格式
- `group_id` 可以从 `ServerID` 的前4位解析得到

#### 2.5.2 配置文件管理

- `resources/excel_tables/` 目录下的配置表
- 统一使用Excel格式管理所有游戏数据配置
- 配置表应包含版本号/更新日期等信息

#### 2.5.3 配置加载流程

- `GameServer` 启动时加载配置表 -> 验证配置
- `MapServer` 启动时调用 `LoadAllTables()` -> 验证配置
- `MapServer` 的 `ServerID` 配置应与 Gateway/Game 保持一致

#### 2.5.4 集成测试

- 运行 smoke 测试 + 配置验证
  `powershell -ExecutionPolicy Bypass -File .\scripts\smoke-integration.ps1`

- 运行客户端 smoke 测试
  `powershell -ExecutionPolicy Bypass -File .\scripts\smoke-integration.ps1 -RunClientSmoke`

## 三、日志管理

### 3.1 日志初始化
- 使用 `zLog.InitLogger()` 初始化日志
- 日志文件应放置在 `./logs/` 目录
- 日志级别应根据环境设置

### 3.2 日志使用
- 使用 `zLog.Info()`, `zLog.Error()`, `zLog.Fatal()` 等方法记录日志
- 日志应包含足够的上下文信息
- 避免在日志中记录敏感信息

## 四、错误处理

### 4.1 错误传递
- 错误应向上传递，不被忽略
- 错误传递时应添加上下文信息
- 使用 `fmt.Errorf("...: %w", err)` 包装错误

### 4.2 错误处理
- 服务器启动时的错误应导致启动失败
- 运行时错误应被捕获并记录，不影响服务器运行
- 关键错误应触发告警

## 五、最佳实践

### 5.1 代码组织
- 每个文件应保持单一职责
- 函数应保持简短，不超过50行
- 避免重复代码，提取公共功能到共享包

### 5.2 性能优化
- 避免频繁的内存分配
- 使用对象池复用对象
- 合理使用并发

### 5.3 安全性
- 验证所有用户输入
- 防止 SQL 注入和 XSS 攻击
- 敏感数据应加密存储

### 5.4 可维护性
- 编写清晰的注释和文档
- 使用一致的命名和代码风格
- 编写单元测试和集成测试

## 六、总结

统一的目录结构和配置管理规范有助于提高代码的可读性、可维护性和可扩展性。所有开发人员应严格遵循本规范，确保项目代码的一致性和质量。

随着项目的发展，本规范可能会根据实际需求进行调整和完善。
